# Exoclaw System Prompts

This file inventories the prompt text that shapes Exoclaw model behavior and
where each piece is managed.

## Always-Included Prompts

### Identity Prompt

Path: `examples/exoclaw/prompts/me.md`

Loaded by: `examples/exoclaw/harness.ts`

Role: `developer`

Purpose: Defines Exoclaw's identity as a long-running local control agent and
sets general operating rules for adapters, scheduled work, setup uncertainty,
sandbox use, and concise operational replies.

This prompt is included on every Exoclaw model call, including normal REPL
turns, adapter wakeups, and scheduled task wakeups. It is committed to git and
must stay generic: do not put a specific user's name, deployment secrets, phone
numbers, account names, or local preferences here.

### Local Profile Prompt

Default path: `.exo/exoclaw-profile.md`

Override path: `EXOCLAW_LOCAL_PROMPT_FILE`

Created by: `examples/exoclaw/scripts/exoclaw-repl setup-profile` or
`examples/exoclaw/scripts/exoclaw-repl --setup-profile`

Loaded by: `examples/exoclaw/harness.ts`

Role: `developer`

Purpose: Holds local, deployment-specific details such as the user's name or
extra operating preferences. This file is additive to the generic identity
prompt and should stay out of git. The default path lives under `.exo`, which is
already ignored.

Use this prompt for local identity and deployment details, for example:

- the user's preferred name
- preferred response style for this machine
- local adapter/channel conventions that are not generally useful
- temporary testing instructions for a local Exoclaw instance

Create or update it interactively with:

```bash
examples/exoclaw/scripts/exoclaw-repl setup-profile
```

For a fresh Exoclaw build that also prompts for the local profile before setting
up all adapters:

```bash
PATH="/opt/homebrew/opt/openjdk/bin:$PATH" \
  examples/exoclaw/scripts/exoclaw-repl fresh \
  --agent spooky \
  --agent-name Spooky \
  --conversation dev \
  --setup-profile \
  --setup-all
```

If a deployment wants the local prompt somewhere other than `.exo`, set
`EXOCLAW_LOCAL_PROMPT_FILE` or pass `--local-prompt-file <path>` to the launcher.

### Harness Operations Prompt

Path: `examples/exoclaw/harness.ts`

Role: `developer`

Purpose: Documents Exoclaw-specific tool behavior: scheduling tools, adapter
tools, delete vs. disable semantics, sandbox defaults, and external reply
boundaries.

This prompt is currently inline in source code. It should probably move to a
Markdown file such as `examples/exoclaw/prompts/operations.md` if it continues
to grow.

### Agent Tool Creation Prompt

Path: `examples/typescript/turn-loop.ts`

Role: `developer`

Purpose: Teaches the model how to create reusable tools with
`install_agent_tool`.

This prompt is included only when `enable_agent_tool_creation` is true for the
agent.

### Persisted Agent Instructions

Path: stored in the agent config as `agentConfig.instructions`

Role: whatever each stored message specifies

Purpose: General per-agent instructions available to all harnesses. Exoclaw
currently uses the harness-level identity prompt instead of requiring persisted
agent instructions.

## Startup Prompts

Startup prompts are sent as user messages by `examples/exoclaw/scripts/exoclaw-repl`.
They affect the conversation history but are not automatically injected on every
future model call.

### Adapter Setup Prompts

Paths:

- `examples/exoclaw/adapters/irc/setup-prompt.md`
- `examples/exoclaw/adapters/whatsapp/setup-prompt.md`
- `examples/exoclaw/adapters/signal/setup-prompt.md`

Sent by: `examples/exoclaw/scripts/exoclaw-repl --setup <adapter>`

Purpose: Ask Exoclaw to create or verify a specific adapter and explain manual
setup steps such as QR pairing or IRC test instructions.

### Initial Prompt File

Path: user-provided via `--initial-prompt-file <path>`

Sent by: `examples/exoclaw/scripts/exoclaw-repl`

Purpose: Allows test or demo instructions to be supplied from the host
filesystem before the interactive REPL starts.

## Runtime Wakeup Prompts

Runtime wakeup prompts are generated by Rust and sent as user messages when
external or scheduled events occur.

### Adapter Wakeups

Path: `crates/executor/src/adapter/runtime.rs`

Role: user message

Purpose: Converts an inbound adapter message into an Exoclaw turn. The prompt
includes adapter type, target, sender, message text, adapter id, and instructions
for replying with `send_adapter_message`.

### Scheduled Task Wakeups

Path: `crates/executor/src/scheduler_runtime.rs`

Role: user message

Purpose: Converts scheduled task completion or failure into an Exoclaw turn. The
prompt includes run metadata, result artifact information, output previews, and
the task's `reportPrompt`.

## Should Prompts Move Out Of Source?

Mostly yes.

Short operational prompts that are tightly coupled to code can stay inline, but
the Exoclaw identity and larger behavior prompts are easier to review, tune, and
test when they live as Markdown files. A good target shape is:

- Keep prompt inventory in this file.
- Keep identity in `examples/exoclaw/prompts/me.md`.
- Move the inline Exoclaw operations prompt from `harness.ts` to a Markdown file
  if it grows or changes often.
- Keep generated wakeup prompt templates near the Rust code that constructs
  their dynamic fields, unless they become large enough to benefit from template
  files.
