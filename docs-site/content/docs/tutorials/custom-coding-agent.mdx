This tutorial builds a small **coding agent**: one that reads and edits code
in a sandbox, runs commands, and works through a multi-step task on its own.
It reuses the pieces from the
[Custom Agent Quickstart](./write-your-own-agent.mdx) — a `registerTools`
hook and an `instructions` hook — and adds the two things a coding agent
specifically needs: a **good-but-simple system prompt** and **task
tracking** so the model stays coherent across many steps.

The design here follows what production coding agents like
[opencode](https://github.com/sst/opencode) converged on, distilled to the
minimum. The finished file is
[`examples/typescript/coding-agent-harness.ts`](https://github.com/ankrgyl/exo/blob/main/examples/typescript/coding-agent-harness.ts).

## You don't write the loop

A coding agent needs to take many steps per request — read a file, run a
test, edit, re-run — before it's done. That "keep going while the model
wants to act" loop is **already built into exo**.
`runResponsesHarnessTurn` calls the model, executes any tool calls it
makes, appends the results, and calls the model again — repeating until the
model stops requesting tools (bounded by the agent's `maxToolRoundTrips`).

So multi-turn reasoning is free. Your job is three things: the prompt, the
tools, and keeping the model oriented across all those steps.

## The system prompt: short and opinionated

You do **not** need a giant prompt, and you do not need a different prompt
per model. One concise prompt that states the role, demands brevity, and
gives a few working rules goes a long way:

```ts
const SYSTEM_PROMPT = `You are a coding agent working from a terminal. You have a shell tool that runs commands in a sandbox and a todowrite tool for tracking multi-step work.

Be concise and direct; this is a CLI, so keep prose short and let tool calls do the work. Prioritize technical accuracy over agreement.

Working style:
- For any task with 3 or more steps, call todowrite first to lay out a plan, then keep it updated as you go. Keep exactly one item in_progress at a time. Mark an item completed only after you have verified it works — a command that ran, a test that passed — never on intent alone.
- Inspect the codebase with commands (ls, cat, grep) instead of guessing.
- When you finish, verify your work by running it.
- Reference code locations as file_path:line_number so they are easy to find.`;
```

The transferable techniques packed into those few lines:

- **State the role and the medium first** ("coding agent … from a
  terminal") — it primes short, CLI-appropriate output.
- **Explicitly demand conciseness.** Models default to verbose; counteract
  it.
- **An objectivity clause** ("technical accuracy over agreement") reduces
  sycophancy.
- **Tell it to inspect, not guess**, and to **verify by running**.
- **`file_path:line_number`** makes answers navigable in a terminal.

Keep detailed per-tool rules in the *tool descriptions*, not the system
prompt — that keeps the prompt small and the rules next to the tool they
govern.

### Grounding context, generated each round

Alongside the static prompt, inject a small block of live context. This is
the [context-building](./write-your-own-agent.mdx) hook again — cheap,
high-value grounding, rebuilt every round:

```ts
{
  role: "developer",
  content:
    `<env>\n` +
    `Platform: ${os.platform()}\n` +
    `Today's date: ${new Date().toDateString()}\n` +
    `Files live in the sandbox; use the shell tool to read and change them.\n` +
    `</env>`,
}
```

Note it states *where the files are* rather than dumping a file tree — the
agent is expected to run tools to explore. That keeps the prompt small.

## The minimal tool set

A coding agent needs surprisingly few tools. The single most important one
is **shell**, and exo ships it as a built-in that runs inside the
conversation's sandbox. With shell alone the model can read (`cat`), search
(`grep`), edit (`sed`), build, test, and run:

```ts
async function registerCodingTools(
  tools: HarnessToolRegistry,
  context: TurnContext,
): Promise<void> {
  registerBuiltInTools(tools, context, ["shell"]);
  await registerLibraryTools(tools, context, todoTool); // defined below
}
```

<Callout type="info">
  **What a fuller agent adds.** Production agents wrap common operations in
  dedicated tools — a line-numbered `read`, and an `edit` that does an exact
  string replacement and **fails if the target string isn't unique**. That
  uniqueness guard is the key design choice: it makes edits deterministic and
  prevents silent wrong-location changes. They're worth adding, but they're
  refinements on top of `shell`, not prerequisites — so we leave them out of
  the minimal build.
</Callout>

## Task tracking: a todo tool

For any multi-step job, the model needs to remember the plan and where it
is in it. The pattern that works: a single `todowrite` tool where the model
**rewrites the entire list** each call, with a small status enum, and the
current list **re-surfaced to the model every round**.

The tool stores the list as a conversation
[artifact](../concepts/data-model.mdx) so it survives across turns:

```ts
const todoTool = defineTool({
  definition: {
    name: "todowrite",
    description:
      "Record and update your task list. Rewrite the FULL list on every call. " +
      "Use it for any task needing 3 or more steps; skip it for trivial one-step " +
      "work. Keep exactly one item in_progress, and mark an item completed only " +
      "after you have verified it is actually done.",
    parameters: {
      type: "object",
      additionalProperties: false,
      properties: {
        todos: {
          type: "array",
          description: "The full, updated task list.",
          items: {
            type: "object",
            additionalProperties: false,
            properties: {
              content: { type: "string" },
              status: {
                type: "string",
                enum: ["pending", "in_progress", "completed", "cancelled"],
              },
            },
            required: ["content", "status"],
          },
        },
      },
      required: ["todos"],
    },
  },
  initializationParameters: {
    type: "object",
    additionalProperties: false,
    properties: {},
  },
  initialize() {
    return {
      async execute(args, execution) {
        const todos = (args.todos ?? []) as unknown as Todo[];
        await writeTodos(
          execution.context.exoharness.current.conversation,
          todos,
        );
        const remaining = todos.filter(
          (todo) => todo.status !== "completed" && todo.status !== "cancelled",
        ).length;
        return { ok: true, remaining };
      },
    };
  },
});
```

Three design choices carry most of the value:

- **Full-list rewrite, not incremental patches.** The model resends the
  whole array; there's no id bookkeeping in the loop, and the tool is
  stateless.
- **Exactly one `in_progress`.** This forces linear focus and gives a clear
  "what's happening now" signal.
- **"Completed only after verification, never on intent."** The single most
  important anti-hallucination rule for task tracking — stated in both the
  tool description and the system prompt.

### Re-surfacing the list each round

Storing the list isn't enough; the model needs to *see* it on every step.
Read it back in the `instructions` hook and append it — the same per-round
injection you used for telemetry in the previous tutorial:

```ts
const todos = await readTodos(conversation);
if (todos.length > 0) {
  const rendered = todos.map((t) => `- [${t.status}] ${t.content}`).join("\n");
  messages.push({
    role: "developer",
    content: `Your current task list (from todowrite):\n${rendered}`,
  });
}
```

The artifact helpers are small — write to a fixed path, read the latest
version back:

```ts
const TODO_ARTIFACT_PATH = "coding-agent/todos.json";

async function readTodos(conversation: Conversation): Promise<Todo[]> {
  const latest = (await conversation.listArtifacts())
    .filter((v) => v.path === TODO_ARTIFACT_PATH)
    .sort((a, b) => b.version - a.version)[0];
  if (!latest) return [];
  return (
    (await conversation.readArtifactJson<Todo[]>({
      artifactId: latest.artifactId,
    })) ?? []
  );
}

async function writeTodos(conversation: Conversation, todos: Todo[]) {
  await conversation.writeArtifactJson({
    path: TODO_ARTIFACT_PATH,
    value: todos as unknown as JsonValue,
  });
}
```

## Put it together

```ts
export default defineHarness({
  async runTurn(context) {
    await runResponsesHarnessTurn(context, {
      instructions: codingInstructions,
      registerTools: registerCodingTools,
    });
  },
});
```

## Run it

```bash
exo --harness typescript agent create "Coder" \
  --module examples/typescript/coding-agent-harness.ts \
  --model gpt-5.5
exo conversation create coder "Build"
exo repl --agent coder --conversation coder-build
```

A real run (via `conversation send`), asking it to write a buggy
`count.sh`, find the bug, and fix it — abbreviated to show the shape:

```text
user: write count.sh that prints 1..15 but with an off-by-one that stops
      at 14; plan with todowrite, run it, fix it, run again to confirm.

assistant: [tool_call todowrite]  # 5-item plan, one in_progress
assistant: [tool_call shell] cat > count.sh …   # for ((i=1; i<15; i++))
assistant: [tool_call shell] ./count.sh         # → 1..14, "last=14, lines=14"
assistant: [tool_call todowrite]  # mark "observe bug" done, next in_progress
assistant: [tool_call shell] sed -i 's/i < 15/i <= 15/' count.sh
assistant: [tool_call shell] ./count.sh         # → 1..15, "last=15, lines=15"
assistant: [tool_call todowrite]  # all completed, remaining: 0
assistant: Done. Fixed count.sh at count.sh:3 (i < 15 → i <= 15); final
           run prints 1 through 15.
```

Every model round in that sequence saw the current todo list re-injected,
so the plan stayed intact across all eight-plus steps — and the agent only
marked steps completed after a command confirmed them.

## Where to go next

- Add the dedicated **`read`** and **`edit`** tools described above; the
  unique-match guard on `edit` is the highest-leverage refinement.
- Delegate open-ended exploration to a **sub-agent** so noisy file reads
  stay out of the main context window.
- Set a sensible **`maxToolRoundTrips`** on the agent so a runaway task
  stops cleanly instead of looping.
- Read [Executors & Harnesses](../concepts/executors.mdx) to see how this
  compares to the built-in `codex` / `claude-code` / `cursor` harnesses,
  which wrap full coding-agent runtimes with durable exoharness state.
